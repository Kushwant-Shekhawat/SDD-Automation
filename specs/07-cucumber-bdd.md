# SDD-Automation Framework - Cucumber BDD Specification

## 1. OVERVIEW

All test execution in this framework runs through Cucumber BDD. There are no standalone TestNG test classes. Cucumber uses TestNG as its runner backend via `AbstractTestNGCucumberTests`.

- **Cucumber version**: 7.14.0
- **Dependency injection**: PicoContainer (`cucumber-picocontainer:7.14.0`)
- **Runner**: `CucumberRunner extends AbstractTestNGCucumberTests`
- **Parallelism**: Scenario-level, 4 threads (`@DataProvider(parallel = true)`)

## 4. STEP DEFINITIONS

All step def classes in package `org.example.stepdefs`.

### Dependency Injection via PicoContainer

`SharedContext` is the DI container. PicoContainer creates one instance per scenario and injects it into every step class that declares it in the constructor:

```java
public class LoginSteps {
    private final SharedContext ctx;

    public LoginSteps(SharedContext ctx) {
        this.ctx = ctx;
    }
    // @Given/@When/@Then methods use ctx.loginPage, ctx.page etc.
}
```

`SharedContext` initialises the browser in its constructor and exposes all page objects:
```java
public SharedContext() {
    WebDriverManager.initBrowser();
    this.page = WebDriverManager.getPage();
    this.loginPage = new LoginPage(page);
    // ... other pages
    page.navigate(ConfigReader.getBaseUrl());
}
```

### Step definition classes

| Class | Covers |
|-------|--------|
| `LoginSteps.java` | Login, invalid login, locked user |
| `ProductSteps.java` | Product listing, add to cart, cart badge |
| `CartSteps.java` | Cart view, remove items, continue shopping, totals |
| `CheckoutSteps.java` | Checkout step 1, step 2, confirmation, cancel |
| `NavigationSteps.java` | Menu open/close, logout, reset app state, back navigation |
| `VisualSteps.java` | Visual screenshot comparison (`@visual` only) |
| `Hooks.java` | `@Before`, `@AfterStep`, `@After` lifecycle |

## 5. HOOKS

`Hooks.java` in `org.example.stepdefs`.

### `@Before`
```java
ScreenshotUtil.initScenarioFolder(scenario.getName());
```

### `@AfterStep`
```java
String path = ScreenshotUtil.captureStepScreenshot(ctx.page);
// attach to Extent report via MediaEntityBuilder (relative path)
// attach to Cucumber JSON via scenario.attach(bytes, "image/png", name)
```

### `@After`
```java
if (scenario.isFailed()) {
    String path = ScreenshotUtil.captureFailureScreenshot(ctx.page, scenario.getName());
    // attach to both reports
}
ScreenshotUtil.cleanupScenario();
ctx.tearDown();  // closes browser
```

## 6. CUCUMBER RUNNER

```java
@CucumberOptions(
    features = "src/test/resources/features",
    glue = "org.example.stepdefs",
    plugin = {
        "pretty",
        "json:build/reports/cucumber/cucumber-report.json"
    },
    monochrome = true
)
public class CucumberRunner extends AbstractTestNGCucumberTests {

    @Override
    @DataProvider(parallel = true)
    public Object[][] scenarios() {
        return super.scenarios();
    }
}
```

**Notes:**
- No `html:` plugin — it causes XSS errors and 76 MB files. HTML is generated by Masterthought from the JSON.
- `glue = "org.example.stepdefs"` — exact package name, not `stepdefinitions`
- `@DataProvider(parallel = true)` enables parallel scenario execution

## 7. TESTNG CONFIGURATION

The primary suite file `testng.xml` registers `ExtentTestListener` and runs `CucumberRunner`:

```xml
<suite name="SDD Automation Suite" parallel="none" data-provider-thread-count="4">
  <listeners>
    <listener class-name="org.example.listeners.ExtentTestListener"/>
  </listeners>
  <test name="Cucumber BDD Suite">
    <classes>
      <class name="org.example.runners.CucumberRunner"/>
    </classes>
  </test>
</suite>
```

`parallel="none"` at suite level is correct — parallelism is at the data-provider level inside `CucumberRunner`, controlled by `data-provider-thread-count="4"`.

## 8. RUNNING CUCUMBER TESTS

```bash
# Run all 69 scenarios
./gradlew clean test

# Smoke scenarios only (login, products, checkout happy path)
./gradlew clean test -Dcucumber.filter.tags="@smoke"

# Regression only (excludes @visual)
./gradlew clean test -Dcucumber.filter.tags="@regression"

# Single feature file
./gradlew clean test -Dcucumber.filter.tags="@login"

# Visual regression (local only — baselines are Chromium-specific)
./gradlew clean test -Dcucumber.filter.tags="@visual"

# CI — exclude visual
./gradlew clean test -Dcucumber.filter.tags="not @visual"
```

## 10. ADDING NEW SCENARIOS

1. Add Gherkin in the relevant `.feature` file (or create a new file)
2. Run `./gradlew test` — unimplemented steps will print as `UNDEFINED` with a snippet
3. Copy the snippet into the appropriate step def class in `org.example.stepdefs`
4. Implement the step using page objects via `ctx`
5. Tag the scenario with `@regression` (and `@smoke` if it's critical path)

---

## 11. BEST PRACTICES

- All step classes receive `SharedContext ctx` via constructor (PicoContainer DI)
- Never create `new LoginPage(page)` directly in step definitions — use `ctx.loginPage`
- Never call `WebDriverManager.initBrowser()` in step defs — `SharedContext` constructor does it
- Keep step definitions thin: call page object methods, assert results — no Playwright API calls
- Step text should express *business intent*, not implementation detail
